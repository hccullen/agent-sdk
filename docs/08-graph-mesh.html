08 — Graph mesh

Everything a "workflow + fan-out + routing graph" DSL would give you, expressed with a single stateGraph() plus a parallel() node. Three researchers fan out, a planner decides whether to gather more evidence, a drafter writes with retries, and a reviewer loops until the draft is approved. 

// Diagram
//
//   gather ──► plan ──(needs_more?)──► gather       (bounded by maxIterations)
//                 │
//                 └──► draft ──► review ──(ok?)──► END
//                                  │
//                                  └──► draft       (agentNode retries)

Run it

npm run mesh

Fan-out as a node

A Parallel is itself a Runnable, so it composes directly inside an agentNode(). The three branches run concurrently; their reply-message parts are concatenated into one MessageResponse, which the merge callback captures into state. 

.addNode("gather", agentNode(
  parallel([pubmed, textbooks, guidelines]),
  (s) => s.topic,
  (r, s) => ({
    ...s,
    evidence: [...s.evidence, ...(r.statusMessage?.parts ?? [])],
  }),
))
Nothing special happens at the node level — as far as the graph is concerned, gather is just a node that calls .run(). The fan-out lives entirely inside the Parallel, and the merge of parts is handled by Parallel.run().

Re-gather loop and retry loop

Two independent loops live in this graph, both bounded by maxIterations on the outer run:

  • Planning loop — if the planner says "more", the plan → gather edge sends execution back for another pass. Each pass appends additional parts to state.evidence.
  • Review loop — if the reviewer rejects, the review → draft edge re-runs the drafter. Independent of the drafter's own per-node retries (which fire only on status === "failed").
.addEdge("plan",   (s) => s.plan.includes("sufficient") ? "draft" : "gather")
.addEdge("review", (s) => s.approved ? END : "draft");
